# ZZUMC 网站二期后端改造分工案

这份文档用于给开发群统一讨论网站二期后端建设范围、技术栈、模块拆分、分工方式和排期。目标不是重做现有主站，而是在现有网站基础上补齐后台管理、数据库、内容发布、资源管理、服务器信息同步、皮肤站登录和管理员权限体系。

## 1. 项目背景

目前主站已经完成了基础前端页面，技术栈是 `Vue 3 + Vite + Vue Router`。从现有仓库情况来看：

- 新闻页面目前主要读取本地 `src/content/news` 下的 Markdown 文件。
- 帮助文档页面目前主要读取本地 `src/content/docs` 下的 Markdown 文件。
- 资源下载页面目前主要读取本地 `src/data/downloads.js` 和 `src/lib/downloadsSource.js`。
- 服务器信息页面目前主要由前端直接请求状态接口并使用本地数组配置服务器。
- 关于页面中的成员、赞助者、联系方式等内容仍然是页面内静态数组。

这意味着网站当前适合静态展示，但不适合长期运营。后续如果要支持社团成员自己发新闻、写文档、发资源、维护服务器列表和同步皮肤站用户，就必须增加后端系统。

## 2. 二期建设目标

本期目标如下：

- 新闻支持后台发布、编辑、删除、上下架、置顶和首页展示位管理。
- 帮助文档支持后台新建、分类、排序、编辑和发布，正文继续使用 Markdown。
- 资源下载改成数据库驱动，文件位置和元数据由后端管理，前台自动获取可下载资源。
- 服务器信息支持后台添加和修改服务器，前台自动读取服务器状态、在线人数、在线玩家名称和玩家头颅。
- 关于页面支持从皮肤站接口自动同步所有已注册玩家，不再手动维护成员列表。
- 网站支持使用皮肤站账号登录，不额外维护一套独立网站账号密码。
- 对指定成员开放管理员身份，用于发布新闻、编写文档、发布资源和维护服务器信息。
- 管理员权限需要区分不同职责，避免所有人都拿最高权限。

## 3. 总体设计思路

总体方案是保留现有主站前端结构，把原来读取本地静态数据的部分逐步替换成后端 API。

整体方向如下：

- 主站前端继续保留现有页面结构，改为通过接口获取数据。
- 增加一个独立后端服务，负责数据库、鉴权、业务接口和文件管理。
- 增加一个独立后台管理端，供管理员登录后发布内容和维护数据。
- 与皮肤站做账号打通，优先采用标准接口或单点登录方案。
- 与 Minecraft 服务器状态接口或自定义插件接口联动，拿到更准确的在线玩家信息。

## 4. 推荐技术栈

结合当前主站技术栈和后续维护成本，推荐如下：

### 4.1 前端

- 主站前端：`Vue 3 + Vite + Vue Router`
- 管理后台：`Vue 3 + Vite + Element Plus + Pinia`

说明：

- 主站不需要重写，只需要把数据源从本地文件切换成接口。
- 后台管理端建议单独做 `/admin`，不要和主站页面混在一起。
- 管理后台优先追求开发效率和稳定性，不需要和官网首页保持同样的视觉风格。

### 4.2 后端

- 运行环境：`Node.js 20`
- 后端框架：`NestJS`
- ORM：`Prisma`
- 数据库：`MySQL 8`
- 鉴权：`JWT + RBAC`
- 接口文档：`Swagger / OpenAPI`

说明：

- `NestJS` 比较适合多人协作、模块拆分和后期维护。
- `Prisma` 适合快速定义表结构和迁移，类型安全也更好。
- `RBAC` 用于做后台角色和权限控制。

### 4.3 文件与部署

- 文件上传：MVP 阶段可先用服务器本地 `uploads/`
- 后续扩展：可迁移到 `MinIO` 或云对象存储
- 部署方案：`Docker Compose + Nginx + MySQL`

说明：

- 下载资源和新闻封面图不建议直接混在前端静态资源目录里。
- 数据库存储文件元数据，文件本体放存储目录或对象存储。

## 5. 核心业务需求拆分

### 5.1 最新消息模块

需求：

- 管理员可以登录后台发布新闻。
- 新闻支持标题、摘要、封面图、正文、标签、作者、发布时间。
- 新闻支持草稿、已发布、已下线三种状态。
- 新闻支持首页推荐位和新闻页展示排序。
- 新闻正文继续使用 Markdown，便于和现有渲染逻辑兼容。

后台功能：

- 新闻列表
- 新建新闻
- 编辑新闻
- 删除新闻
- 发布/下线
- 设置推荐位
- 封面上传

前台接口：

- 获取首页新闻列表
- 获取新闻页列表
- 获取新闻详情
- 获取标签列表

### 5.2 帮助文档模块

需求：

- 管理员可以在后台创建帮助文档。
- 文档支持分类、标题、描述、正文、排序、更新时间。
- 文档正文继续使用 Markdown。
- 文档支持目录导航和页面内标题导航。

后台功能：

- 文档分类管理
- 文档列表
- 新建文档
- 编辑文档
- 删除文档
- 发布/下线
- 排序调整

前台接口：

- 获取文档导航树
- 获取首篇文档 slug
- 获取文档详情

### 5.3 资源下载模块

需求：

- 下载资源不再写死在前端数组中。
- 资源信息由数据库统一管理。
- 文件上传后由后端记录存储路径、版本、大小、分类和下载链接。
- 前台自动读取资源列表并生成下载按钮。
- 后台支持推荐资源位。

后台功能：

- 资源分类管理
- 文件上传
- 资源信息编辑
- 资源发布/下线
- 设置推荐资源
- 统计下载次数

前台接口：

- 获取下载页数据
- 获取资源分类
- 获取推荐资源
- 下载文件接口

实现要求：

- 不建议前台直接暴露真实磁盘路径。
- 下载应统一经过后端接口，便于做权限、统计和后期迁移存储。

### 5.4 服务器信息模块

需求：

- 后台可以维护多个服务器节点。
- 前台显示服务器名称、类型、封面、地址、端口、在线状态、版本、MOTD、在线人数。
- 能展示当前在线玩家名字和玩家头颅。
- 支持手动启用或停用某个服务器展示。

后台功能：

- 新增服务器
- 编辑服务器
- 删除服务器
- 启用/停用服务器
- 手动刷新状态

前台接口：

- 获取服务器列表
- 获取服务器状态
- 获取在线玩家列表

实现要求：

- 普通公开 ping 接口不一定总能返回完整在线玩家列表。
- 如果要稳定显示玩家名称和头颅，最好额外做服务端代理接口，或在服务端安装轻量插件，把在线玩家列表提供给网站后端。
- 网站前端不应直接承担所有轮询逻辑，建议由后端定时采集并缓存，前台只读缓存结果。

### 5.5 关于页面与玩家同步模块

需求：

- 关于页面中的成员相关数据改成由皮肤站接口自动同步。
- 自动获取所有已注册玩家。
- 可按是否展示、是否核心成员、是否赞助者、是否管理员等字段筛选展示。

后台功能：

- 同步皮肤站玩家数据
- 标记展示状态
- 标记身份标签
- 手动刷新同步

前台接口：

- 获取社区成员列表
- 获取核心成员列表
- 获取特别感谢/赞助者列表

实现要求：

- 不建议把皮肤站数据库直接暴露给主站。
- 最好由后端定时同步到本地缓存表，再对前台提供统一接口。

### 5.6 皮肤站登录与后台权限模块

需求：

- 主站支持使用皮肤站账号登录。
- 后台管理员身份基于皮肤站账号授予，不单独注册后台账户。
- 支持不同管理员权限分级。

建议角色：

- 超级管理员：拥有所有权限。
- 内容管理员：可发布新闻和文档。
- 资源管理员：可管理下载资源。
- 服务器管理员：可维护服务器信息。

后台功能：

- 登录/退出
- 用户信息读取
- 角色分配
- 权限校验
- 后台访问控制

实现要求：

- 优先对接皮肤站标准登录接口、OAuth2、JWT 或单点登录。
- 如果皮肤站没有成熟 SSO，需要尽早确认是否能提供可信登录票据或回调接口。
- 权限校验必须由后端执行，前端隐藏按钮不算安全。

## 6. 推荐后端模块划分

后端建议拆成以下模块：

- 认证模块：处理皮肤站登录、JWT 签发、用户信息同步。
- 权限模块：处理角色、权限点和接口鉴权。
- 新闻模块：处理新闻 CRUD、推荐位、发布状态。
- 文档模块：处理帮助文档分类、内容、排序和发布。
- 资源模块：处理文件上传、资源信息、下载分发和统计。
- 服务器模块：处理服务器配置、状态轮询、在线玩家缓存。
- 社区成员模块：处理皮肤站注册玩家同步和关于页成员展示。
- 操作日志模块：记录管理员做过的操作。

## 7. 推荐数据库表设计

初步建议如下：

- `users`
- `roles`
- `user_roles`
- `news_posts`
- `news_tags`
- `news_post_tags`
- `doc_categories`
- `doc_articles`
- `resource_categories`
- `resources`
- `servers`
- `server_status_cache`
- `member_cache`
- `audit_logs`

字段方向建议：

### 7.1 用户与权限

- `users`：本地用户表，保存皮肤站用户 ID、用户名、UUID、头像、最近登录时间
- `roles`：角色定义
- `user_roles`：用户和角色关联

### 7.2 新闻

- `news_posts`：标题、摘要、正文 Markdown、封面、作者、发布时间、状态、推荐位、排序
- `news_tags`：标签
- `news_post_tags`：新闻标签关联

### 7.3 文档

- `doc_categories`：文档分组、排序
- `doc_articles`：标题、描述、正文 Markdown、分类、排序、更新时间、状态

### 7.4 下载资源

- `resource_categories`：分类名、排序
- `resources`：标题、摘要、版本、目标版本、文件大小、存储路径、哈希值、是否推荐、状态、下载次数

### 7.5 服务器

- `servers`：服务器名称、地址、端口、类型、描述、封面、标签、排序、启用状态
- `server_status_cache`：在线状态、版本、MOTD、在线人数、最大人数、玩家列表缓存、更新时间

### 7.6 社区成员

- `member_cache`：皮肤站玩家 ID、UUID、用户名、头像、是否展示、是否核心成员、是否赞助者、同步时间

### 7.7 日志

- `audit_logs`：操作人、操作模块、操作类型、操作对象、变更摘要、时间

## 8. 当前仓库迁移建议

因为现在已有静态内容，推荐做一次迁移，而不是推倒重来。

迁移方案如下：

- 把 `src/content/news` 里的 Markdown 解析后导入 `news_posts`。
- 把 `src/content/docs` 里的 Markdown 解析后导入 `doc_articles` 和 `doc_categories`。
- 把 `src/data/downloads.js` 中的数据导入 `resources` 和 `resource_categories`。
- 把现有页面里的静态服务器配置导入 `servers`。

这样做的好处：

- 现有站点内容不会丢。
- 前后端联调时可以马上看到已有数据。
- 开发过程中可以逐步替换数据源，而不是一次性重写。

## 11. 当前最大风险

### 11.1 皮肤站登录接口不明确

如果皮肤站没有提供稳定的登录授权接口，网站登录和管理员授权会成为最大阻塞项。这个问题必须优先确认。

### 11.2 在线玩家列表不稳定

很多公开状态接口只能稳定拿到在线人数，拿不到完整 sample 玩家列表。要展示玩家名字和头颅，可能需要服务端插件或代理层支持。

### 11.3 下载文件路径管理不规范

如果资源下载直接暴露真实存储路径，后面迁移和统计都会很麻烦。必须一开始就统一通过后端下载接口管理。

### 11.4 权限只做前端控制会有安全问题

后台按钮隐藏不能代替后端鉴权。所有写操作接口必须校验管理员身份和权限。

## 13. 最终预期结果

这期做完后，网站应达到以下状态：

- 新闻不再靠改仓库文件发布。
- 帮助文档不再靠手工往 `src/content/docs` 塞 Markdown。
- 资源下载不再靠前端写死地址。
- 服务器列表不再写死在页面里。
- 关于页面成员不再手动维护数组。
- 管理员可以直接登录后台完成日常内容运营。
- 整个网站从静态展示站升级成可持续维护的社团门户。

## 14. 建议下一步

建议立刻确认三件事：

- 皮肤站是否支持标准登录或可提供可信授权接口。
- Minecraft 服务器是否能稳定提供在线玩家列表。
- 开发群最终有几个人可以实际投入开发。

这三件事一旦确认，就可以继续产出下一版文档：
